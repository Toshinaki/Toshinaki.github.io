当サイトについて

なぜこのサイトを作ったのか

昨年（2025年）の2、3月頃、転職活動のために必死に履歴書を送っていた際、CMS構築サービスを提供している会社に偶然出会いました。久しぶりに「CMS」という言葉を耳にし、脳内の記憶が一瞬で呼び起こされました。何年も前にWordpressを使い、今思えば少し「幼稚」だったサイト制作の経験を思い出したのです。

プログラミングを始めてから長年、技術を学ぶために多くのチュートリアルや記事などを読んできました。また、デザインが美しかったり、斬新なアニメーションやパーティクル効果があったりする、目を引くサイトもたくさん見てきました。そのため、自分もこんなサイトを作ってみたい（あるいは、真似してみたい）と突飛なことを考えることが何度かありました。しかし、往々にして理想は素晴らしいものの、実際に作り始めてみると、決して簡単ではないことに気づかされます。

優れたデザインを多く見すぎると、あちこちから少しずつ参考にしようとしてしまい、結果として葛藤や内的な消耗が生じ、大抵は着手した直後に挫折してしまいます。体系的にデザインを学んだことのない素人が、完全で統一感のあるデザインを作り上げるのは、当然ながら不可能です。

ブログプラットフォームのようなサービスを使えばいいのではないかと考えたこともありました。しかし、アカウントを登録しただけで、その後の進展はありませんでした。

今思えば、私の主な目的は、サイトを作った後にそれを使って何かをすることではなく、単にサイトを構築するというプロセスそのものを体験することにあったようです。

月を指す指は月そのものではありませんが、私はいつも無意識に指の方を見てしまうのです。

そこで深く反省した結果、問題はコードのレベルにあるのではなく、コンテンツの企画とビジュアルデザインで詰まっているのだと気づきました。私は実際に何かを始める前に、考えすぎてしまう傾向があります。例えば、タグをいくつ定義すべきか？ カテゴリをどう分けるか？ スタイルのトーンをどう統一するか？ 多言語対応は必要か？ といったことです。その上、私は面倒なことが大嫌いです。プログラミングの目的は、手作業を減らすことにあるはずですから。結局、着工する前にやる気を失ってしまうことが多々ありました。

「完璧であることよりも、完成させることの方が重要である」。そう考え、最終的に今回はコンテンツを重視することに決めました。まずは自分が書き続けられるようにし、それ以外の重要でないことはすべて後回しにしました。

もちろん、最終的な技術スタックを確定し、この記事を書き始めるまでには、それなりに長い葛藤の期間がありました。

（そして今、これはこの記事の第3バージョンです。）

核心的な動機

サイト構築への興味という要素を除けば、最も根本的な原動力は、私自身がコンテンツプラットフォームを必要としていたことにあります。

私は学ぶことが好きですし、自分でも学ぶことが好きだと思っています。しかし、学生時代の名残か、「学んだらノートを取らなければならない、さもなければ学んでいないのと同じだ」という考えをどうしても捨てきれません。一方で、既存のノートプラットフォームやソフトウェアを使うのは好きではなく、あちこちに .txt や .doc が散らばるのも嫌いです。整理が非常に面倒だからです。その結果、ノートを取るのが億劫になり、学習を始めること自体が難しくなっていました。学んだ知識を記録に残さないことは不安を引き起こし、それがまた新しい学習を始めるのを妨げるという、負のスパイラルに陥っていました。

技術選型

第一の原則は、すべてを執筆のために供することです。まずは自分が書き続けられるようにし、それ以外の重要でないことはすべて後回しにします。これは、使いやすいエディタが必要であるだけでなく、怠惰を含む執筆へのあらゆる障害を最大限に取り除けるソリューションを求めていることを意味します。

以前に Next.js + Markdown ドキュメント、および Next.js + CMSとしての Github issues を使用した経験を振り返り、強力な CMS の重要性を認識しました。CMS といえば、経験豊富というわけではありませんが、かつて Wordpress (PHP) や Wagtail (Python) を少しかじったことがあります。

しかし、独学でフロントエンドエンジニアになった身としては、MDX への対応が現在最も重要な考慮事項となりました。軽量さとコストを考慮した結果、最終的に TinaCMS を選びました。欠点はありますが、なんとか使えています。

次はフロントエンド部分です。

React に慣れているため、本能的に React ベースのソリューションを選びたくなりました。しかし、近年の React の発展方向に対する不安や、当時 Astro に触れたこともあり、React エコシステムや UI ライブラリを捨てるのは難しかったものの、当サイトはコンテンツ優先であることを考慮し、初版は最終的に Astro を選択しました。

しかし、Astro の自由度の高さは、自分で手を動かさなければならない部分も多いことを意味します。各種コンポーネントや CSS を手書きするのを避けるため、最終的に Next.js に戻しました。それが第2版と現在のバージョンです。これは私がこのフレームワークを特別好んでいるという意味ではありません（現在の React の問題は、基本的に Next.js の背後にある会社に起因しています）。純粋に React エコシステムの中で SSG を行うための、より良い選択肢がなかったからです。

第3版？

なぜ第3版を作るのか？

第3版と第2版の実装上の違いは、実はそれほど大きくありません。第2版も正常に公開され、しばらく運用されていました。最終的に諦めた理由は、やはり「面倒だったから」です。

当時は生成 AI が大きな話題になっていた時期で、私もコーディングに AI の補助を多用していました。初版と第2版は AI の助けを借りて構築されました。しかし、コードを書けば書くほど乱雑になり、ビジュアルデザインの選択に迷い、複雑な機能の実装は停滞しました。さらに現実的な要因も重なり、第2版は半年ほど放置されていました。

そして2026年1月の現在、AI のコーディング能力はさらに強力になり、第2版の経験も踏まえ、今回は以前踏んだ地雷をすべて避けるように努めています。

多言語対応

サイトには多言語対応を導入しましたが、基本的には中国語での執筆がメインとなり、他の言語は基本的に AI 翻訳に頼っています（AI に感謝）。翻訳の質は保証できませんが、時間がある時に手動で校正するかもしれません。まあ……適当に見ていただければ幸いです。
