为什么会有这个网站为

背景

今年（2025）二三月份换工作、狂投简历的时候遇到一家提供 CMS 建站服务的公司，忽然就想起了之前多次不成功的建站尝试。浅浅回想一番，我发现问题并不出在代码层面，而是卡在内容规划和视觉设计上——我很容易在真正开始做一件事情前就想太多。比如要预先定义多少标签？如何划分分类目录？样式风格怎么统一？是否需要多语言支持？而我又特别怕麻烦。纠结到最后往往还没动工就丧失动力。于是这次痛定思痛，决定以内容为主，首先要让自己能写下去，其他不重要的一律靠后。当然，在最终确定技术栈、开始写这篇文章前，还是度过了不短的一段纠结期。

而现在这已经是这篇文章的第二个版本了。

技术选型

找到工作，在新工作开始前空闲很多，决定“这次一定要好好搭建网站”后，便开始了技术选型。

首要原则是一切为写作服务。这不仅意味着需要顺手的编辑器，更要求最大限度消除包括懒惰在内的一切可能阻碍写作的因素。

回顾之前用 Next.js + Markdown 和 Next.js + Github issues 的失败经历后，我意识到：比起手动管理文档，还是需要引入CMS系统。而说到CMS，虽说不是经验丰富，也曾经浅尝辄止过 Wordpress（PHP）和 Wagtail（Python）。然而作为半个程序员，也受到前几年前端工作的影响，对 MDX 的支持成了最主要的考虑因素。光是这一条就能刷掉大部分的框架了。再结合轻量化和成本考量，最终确定了 CMS 为 TinaCM。有瑕疵，但勉强够用。

然后是前端。出于对 React 的熟悉和前几次的经验，我本能地想选择以 React 为基础的一些解决方案。但说实话 React 这几年的发展方向让我有些不安（具体的后续有机会再讨论吧）。虽然难以割舍 React 生态包括其 UI 库，但考虑到本网站以内容优先，最终敲定前端为 Astro。

当然了，这也已经是上一个版本的决定了。

由于 Astro 的高自由度，相反的需要自己动手的部分就相应地多了起来。为了避免手写各种组件和 CSS，最终我还是选择了 Next.js，也就是现在这个版本了。这并不代表我有多喜欢这个框架，纯粹是因为 React 生态中在 SSG 这方面没有更好的选择了。

核心动力

抛开对建站的兴趣这个因素，最根本的驱动力在于我确实需要一个文字管理平台。

我喜欢学习，也认为自己喜欢学习。学习了就要记笔记，否则就等于没有学习。但我又不喜欢用一些笔记平台或软件，也不喜欢东一个.txt西一个.doc，整理是个大麻烦。于是就懒得记笔记，也就往往难以开始学习。而学过的知识不做一个记录属实容易引发焦虑，而这又会导致难以开展新的学习。恶性循环了属于是。
